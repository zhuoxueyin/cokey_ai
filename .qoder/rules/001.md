# 项目统一开发强制规范
## 一、最高前置铁则：开发前必须优先阅读根目录 agent.md
### 1. 执行强制流程
1. 任何编码任务、需求开发、代码修改、新增功能，**第一步必须完整读取项目根目录 agent.md**；未解析完成 agent.md 禁止生成、修改任何代码文件。
2. 读取后完成信息对齐：
   - 提取项目技术栈、分层架构、目录约束、存量业务逻辑、禁用API、已有公共组件、数据表结构、环境变量、测试命令、禁止修改文件清单。
   - 对比用户需求，标记冲突点、缺失依赖、存量兼容风险，先输出对齐说明再开始编码。
3. 无 agent.md 文件时，先生成标准化 agent.md 基础模板，填充项目基础信息后再开发。
4. 任何全链路改动（前端页面/组件、服务端接口、数据表、业务流程）开发完成后，**同步更新 agent.md**，新增组件、路由、表结构、业务规则、测试脚本，保证文档与代码实时同步。

### 2. 禁止行为
- 跳过 agent.md 直接编写代码；
- 无视存量逻辑重复封装组件、重复新增接口；
- 修改架构/数据表结构但不更新 agent.md；
- 擅自修改 agent.md 中标注「禁止修改」的文件、模块、数据表。

### 3. agent.md 标准必备章节模板
```markdown

## 2. 架构分层约束
前端分层、服务端分层、数据库分层、跨层调用限制规则
## 3. 全链路统一规范

## 4. 存量业务现状

## 5. 数据库约束
表字段规范、索引命名、迁移脚本规则、读写规范、禁止SQL写法

## 二、全链路功能完整性通用总规则
所有需求必须闭环覆盖 **前端交互 → 服务端接口 → 数据库存储 → 参数校验 → 异常兜底 → 配套测试**，缺失任一环节判定为开发不完整，禁止交付。
通用自检标准：
1. 需求场景全覆盖：正向流程、边界值、异常场景、权限控制、分页/筛选/排序、新增/编辑/删除/查询完整逻辑；
2. 链路无断点：前端操作请求接口、参数双重校验、业务逻辑执行、数据落库、查询回显全流程贯通；
3. 存量兼容：改动不破坏原有业务，兼容历史数据、旧接口、现有数据表；
4. 可完整自测：提供本地启动步骤、调试接口、前端访问地址、数据库测试SQL。

## 三、前端开发完整性规范
### 1. 页面与组件约束
1. 优先复用 agent.md 登记的全局/业务组件，禁止重复封装；新建组件必须同步录入 agent.md；
2. 页面强制包含完整模块：布局容器、表单/列表、操作弹窗、加载状态、空数据兜底、错误提示、权限显隐、防抖节流处理；
3. 表单双重校验：前端本地校验（必填、长度、格式、数值范围、联动规则）+ 后端接口校验，缺一不可；
4. 交互闭环：增删改操作后自动刷新列表；成功/失败统一弹窗提示；全局捕获网络超时、401、403、500异常。

### 2. 接口与数据规范
1. 统一使用项目封装请求实例，自动携带Token、统一请求头、全局错误拦截；
2. 接口字段显式映射，复杂数据转换抽离公共工具函数，禁止隐式字段依赖；
3. 统一托管列表、弹窗、筛选、表单状态，分页参数存入路由/本地存储持久化；
4. 路由权限管控：路由信息登记至 agent.md；无权限隐藏入口、拦截非法访问，配置404兜底页面。

### 3. 前端交付必备产出
页面代码、业务组件、接口请求函数、TS类型定义、Mock测试数据、多端适配样式、基础无障碍兼容代码。

## 四、服务端开发完整性规范
### 1. 接口层标准
1. RESTful 路由规范，统一版本前缀 `/api/v1/`，GET查询、POST新增、PUT更新、DELETE删除，禁止动词式URL；
2. 所有接口入参使用DTO/Schema强校验：必填、类型、长度、枚举、自定义业务规则；
3. 全局统一返回体（写入agent.md）
```json
{
  "code": 200,
  "msg": "操作描述文案",
  "data": {},
  "pagination": {}
}
```
4. 分层异常捕获：参数异常、业务自定义异常、数据库异常、第三方调用异常分层处理，不向前端暴露原始堆栈；全局统一错误码管理。

### 2. 业务分层强制规范
分层顺序：Controller → Facade门面层 → Service业务层 → Repository仓储层，禁止跨层调用
1. Controller：仅做路由分发、参数接收、结果封装，无业务逻辑；
2. Facade：统一事务边界、多服务编排、登录身份/权限校验；所有写操作必须包裹事务；
3. Service：纯业务规则、数据组装、复杂计算，不直接操作数据库；
4. Repository：仅负责数据库CRUD，封装复杂查询，禁止拼接原生SQL防注入；
5. 权限前置拦截：所有写接口、敏感查询接口提前校验登录身份+角色权限。

### 3. 日志与第三方调用规范
1. 外部接口、缓存、消息队列调用增加超时、重试、降级、异常捕获；
2. 全链路日志记录：入参、操作人、IP、时间、数据变更、异常完整堆栈；
3. 定时/异步任务：记录执行状态、失败重试机制、异常告警，数据表留存任务执行记录。

### 4. 服务端交付必备产出
接口路由、入参出参DTO、分层业务代码、自定义异常枚举、事务逻辑、单元测试用例、调试CURL脚本。

## 五、数据库开发完整性规范
### 1. 表结构强制标准
1. 每张业务表必备基础字段：`id`主键、`created_at`、`updated_at`、`deleted_at`软删除标记、操作人ID；禁止物理删除业务数据；
2. 命名统一写入agent.md：表名、字段小写下划线；索引命名 `uk_唯一键` / `idx_查询字段`；
3. 字段约束：字符串限定长度、数值设置取值范围、枚举增加注释，所有字段添加中文说明；
4. 关联关系采用代码层逻辑外键校验，梳理一对多/多对一关系，规避循环关联。

### 2. 数据操作约束
1. 新增：必填字段校验、唯一键冲突捕获、关联数据同步创建；
2. 更新：仅更新传入字段，乐观锁处理并发修改，自动刷新更新时间；
3. 删除：统一执行软删除，删除前校验关联数据，禁止直接DELETE语句；
4. 查询：列表接口强制分页，禁止全表无限制查询；高频查询字段建立索引；禁止 `SELECT *`，显式指定查询字段。

### 3. 数据库变更规范
1. 新增表、修改表结构必须生成迁移SQL脚本，存入 `db/migration`，禁止手动修改库表；
2. 迁移脚本兼容存量历史数据，支持回滚，同步更新agent.md；
3. 禁止行为：无索引大表查询、拼接SQL注入风险、批量更新无事务、手机号/身份证等敏感数据明文存储。

### 4. 数据库交付必备产出
迁移建表/改表SQL、索引创建语句、初始化测试数据SQL、仓储CRUD方法、数据唯一性校验逻辑。

## 六、全链路测试完整性规范
开发完成必须覆盖全部测试场景，无测试判定功能不达标：
1. 单元测试：服务端Service、前端工具函数，覆盖所有分支、边界值；
2. 接口测试：正向参数、空参数、非法参数、无权限访问、超大批量数据场景；
3. E2E全链路自测：前端完整操作流程 → 接口请求 → 数据库数据变更核对；
4. 兼容测试：历史存量数据读写、旧接口兼容、并发操作防脏数据；
5. 测试命令、用例统一写入 agent.md，提供一键执行测试脚本。

# 七、交付前置强制复核规则（核心新增：需求不漏检）
## 7.1 交付前不可跳过：用户原始需求逐条核对
1. 固定核对流程
   1）回溯用户完整原始需求文本，逐条拆解为可验证清单；
   2）逐项匹配已完成代码、文档、脚本，标记「已完成/部分完成/未实现/遗漏」；
   3）存在遗漏/部分完成项，立即暂停交付，补齐缺失逻辑；模糊需求主动向用户确认后再开发；
   4）全部需求标记「已完成」，才可进入最终交付环节。
2. 核对覆盖三类需求
   - 显性需求：用户明确提出的页面、接口、数据表、交互、业务功能；
   - 隐性配套需求：隐含校验、异常兜底、分页、权限、日志、迁移脚本、测试用例；
   - 约束需求：用户指定技术栈、禁用写法、目录规范、agent.md约束条款。
3. 严格禁止行为
   - 主观删减用户需求、省略次要配套功能；
   - 不做逐条核对直接交付，造成功能遗漏；
   - 模糊需求自行脑补实现，不主动确认。

## 7.2 交付配套输出要求
交付时附带《需求核对清单》，包含：
- 原始需求条目原文
- 对应实现文件/代码位置
- 完成状态
无法实现、需补充信息的需求单独标注并说明原因，不得隐瞒遗漏。

## 7.3 遗漏兜底流程
复核发现需求遗漏 → 暂停交付 → 补齐全部缺失功能 → 重新执行全链路自测 + 二次需求复核 → 确认无遗漏后再交付。
